# Stakeholder Review Workbook — Design

## Purpose

Every score dast-bench has produced so far comes from one voice: the
evaluator driving the CLI/skills, backed by research and (for two
vendors) hands-on scan data. There is no mechanism for the humans who
actually have to live with the choice — dev-team leads, DAST SMEs on
specific products, security architects with differing priorities — to
weigh in. This feature adds that mechanism, scoped narrowly: a workbook
that lets a panel of stakeholders confirm or dispute the automated
scoring, and feed into the shortlist/finalist decision itself rather
than only rubber-stamp one already made, without building any new
application, authentication system, or ongoing parallel UI. It is
explicitly a **one-time-per-evaluation** feature, not infrastructure
meant to run continuously.

## Scope & trigger

Applies to every vendor with a **complete paper score for every
criterion** — that's it. No finalist/rejected/evaluated status is
required. `dast-shortlist` already guarantees every candidate is fully
paper-scored before any verdict is made (its own "no gaps" check runs
before the finalist recommendation step), so this precondition is
already satisfied for the whole candidate set at that point — including
candidates the evaluator might otherwise reject on non-merit grounds
(e.g. a practical/access limitation, not a capability gap).

This is deliberate: restricting the workbook to only already-declared
finalists would mean the automated pipeline alone decides who a
stakeholder panel even gets to see, before any human judgment enters the
picture. Generating it earlier — covering the whole paper-scored
candidate set — lets the panel's input **inform** the finalist/rejected
decision instead of following it. Whether to trigger before or after
that verdict is the driver's judgment call, not something the tool
enforces.

The workbook is generated by an explicit, manual decision from the
evaluator running the tool (the "driver") — once they've decided which
vendors belong in this round. It is not triggered automatically.

**Each `generate` invocation starts a standalone workbook.** If a
genuinely new vendor is added later (say, a shelved candidate like
Checkmarx gets onboarded and scored) and the driver wants it in the
review, that requires a fresh `generate` covering the new full vendor
set — with no versioning, diffing, or automatic reconciliation against
an earlier generated workbook. If the driver wants to carry information
from an old workbook into a new one, that's a manual step outside the
tool's scope.

This is distinct from **Round 2 populate** (below), which is not a new
workbook at all — it's a deliberate, designed-for continuation of the
*same* workbook, for the specific case of hands-on `dast-scan` results
landing after Round 1 already started.

## Two rounds: paper review, then hands-on results landing

Some criteria — in practice, the ones `dast-scan` hands-on testing
refines (`detection-accuracy`, `false-positive-rate`, and any other
criterion the driver expects hands-on results to meaningfully change) —
have a paper score in Round 1, but the driver may not want stakeholders
rating them against paper-only evidence when a stronger, hands-on number
is expected soon. The driver marks these as **pending** for a given
`generate` invocation (a `--pending-criteria` flag, per vendor if it
varies by vendor). This does not remove the row or change its position
in the priority ordering — it changes what's shown and whether it's
editable:

- The Automated Score/Evidence/Confidence cells show placeholder text:
  *"Pending — dast-scan results not yet available. Do not edit; will be
  populated in Round 2."*
- The stakeholder score/dispute/rationale cells for that row are
  Excel-locked (native sheet protection on just those cells), so nobody
  can rate a row that has no real evidence behind it yet.
- Every other row (non-pending) behaves exactly as already
  designed — full automated evidence shown, stakeholder columns open.

**Round 2 ("populate")** happens whenever `dast-scan` results land for a
given vendor — via the existing, unchanged `dast-bench scan
ingest-scan-result` path into `data/candidates/<id>.yaml`. A new,
narrowly-scoped `populate` command then opens the **same, already
in-flight workbook file, in place** (never a fresh `generate`) and:

- Looks up that vendor's now-current hands-on-refined scores for exactly
  the criteria marked pending for that vendor.
- Writes the real score/evidence into just those placeholder cells,
  removes the cell lock, and clears the placeholder formatting.
- Touches nothing else — every cell a stakeholder has already filled in,
  anywhere in the workbook, is left completely untouched. Round 2 only
  ever fills in cells that were genuinely empty (locked placeholders) in
  Round 1; it never overwrites or re-derives a cell that was ever open
  for editing.

Results land unevenly in practice — one vendor's `dast-scan` results
might be ready long before another's. `populate` is scoped to one vendor
per invocation for exactly this reason; there is no "populate everything
at once" requirement.

**Rows must be addressable by `criterion_id` (and vendor/sheet), not
visual row position** — a stakeholder may have sorted, filtered, or
resized the sheet by the time `populate` runs. `generate` writes a
hidden helper column carrying each row's `criterion_id` specifically so
`populate` (and `merge`, and `validate`) can reliably find the right row
regardless of how the visible sheet has been rearranged.

## Non-goals

- No new web application, login system, or continuously-running service.
  This relies entirely on collaboration tooling the org already has
  (Microsoft 365 / OneDrive / SharePoint co-authoring, or an equivalent
  like Google Sheets) — dast-bench never hosts or serves the workbook
  itself.
- No feeding stakeholder input back into `data/candidates/*.yaml`, the
  weighted-total calculation, or `dast-bench render`'s deterministic
  output. This workbook is a parallel, human-facing artifact — the
  existing evidence-based scorecard is never diluted or overwritten by
  subjective input.
- No reconciliation between two **separately generated** workbooks
  (e.g. a new vendor added later starts a disconnected round, per
  "Scope & trigger" above). This is distinct from the Round
  2 "populate" mechanism, which is a deliberate, designed-for
  continuation of one workbook's lifecycle, not a reconciliation between
  independent artifacts.
- No structured, queryable "stakeholder-ratings" data model that
  dast-bench itself reads back as data (no new Pydantic models, no new
  YAML the CLI parses). The workbook itself is the sole source of truth
  for stakeholder input, from the moment it's generated onward. The
  `snapshot` command (below) does copy point-in-time `.xlsx` files into
  `data/stakeholder-reviews-archive/` for git provenance — that's an
  inert file copy for the historical record, not a data store dast-bench
  ever parses or reads back from.
- No confidence-scoring layer on the evaluator's own evidence (a related
  idea raised during this brainstorm) — that's a separate future
  enhancement to the automated research process, not part of this
  feature.
- No Excel macros or add-ins. Every computed cell uses portable native
  formulas only, since stakeholders' Excel versions/platforms will vary
  and macros often require explicit trust/enablement that not every
  environment allows.

## Workbook structure

One workbook, one sheet per vendor in scope for this round. Every sheet
has the same column layout:

**Base columns** (populated at generation time, read directly from the
existing `CriteriaTaxonomy` and `Vendor` models — no new schema needed):

| Criterion | Category | Weight | Automated Score | Automated Evidence | Automated Confidence |

Plus one **hidden** column carrying `criterion_id`, used by `merge`,
`validate`, and `populate` to address a row reliably regardless of how
the visible sheet has been sorted/filtered/resized.

**Per-stakeholder columns**, repeated once for each named stakeholder
passed to `generate` (e.g. `--stakeholder "Jane Doe:DAST SME"`):

| `<Stakeholder> Score` | `<Stakeholder> Dispute?` | `<Stakeholder> Rationale` |

Role is required for every stakeholder; name is optional (a driver can
generate a round with role-only labels — e.g. `"DAST SME"` with no
name — if anonymity is preferable for a given round; the schema doesn't
force either choice).

**Resolution columns** (always present, filled in manually by the driver
during/after discussion — never computed):

| Resolved Score | Resolved By | Resolved Timestamp |

**Computed column** (a portable formula, not manual entry):

| Automated vs. Resolved Delta |
`=IF(ISBLANK(<ResolvedScore>),"",<ResolvedScore>-<AutomatedScore>)`

### Row ordering (priority tiering)

Every row for a criterion the vendor has a score for is included —
stakeholders are never required to fill in every row, only what they
have an opinion on, and pending rows keep their normal position (just
locked/placeholder, not omitted). Rows are ordered by a computed default
priority so a time-constrained reviewer sees the highest-value rows
first:

1. Primary sort: criterion **weight**, descending — criteria that matter
   most to the final decision get attention regardless of confidence.
2. Secondary tie-break: criteria where the current score is `≤2.5`, or
   whose `data/research-cache/<vendor>.yaml` entry has
   `reviewed_by_gap_check: true`, sort earlier within their weight
   band — these are exactly the criteria where even the automated
   research process flagged its own uncertainty, so a stakeholder's
   real-world experience is most likely to change the picture.

This ordering is a **default**, not a hard rule — the driver can review
and manually reorder or trim the list before generating a given round's
workbook (a future flag on `generate`, e.g. an explicit criteria-order
override file, is a reasonable extension but not required for v1; v1
just needs the computed default to be correct and the generated file to
be a plain spreadsheet the driver can freely reorder rows in by hand
before distributing).

The top tier (a configurable count, default the top 10) gets a visual
fill-color band distinguishing it from the rest, so the priority is
obvious at a glance even without reading the sort order.

### Scoring math with partial data (pending rows)

Because some rows may be pending in Round 1, the weighted-total and
category-subtotal formulas at the bottom of each sheet only sum across
criteria that currently have real data — never renormalized to look like
a complete 100-point total. Displayed as, e.g., *"66/80 available
points"* (available points = sum of weights for non-pending criteria),
alongside a visible, permanent note: *"Provisional — ranking may shift
once pending dast-scan results land."* The note and the "X/Y" framing
stay in place even after some (but not all) vendors' pending rows have
been populated in Round 2, since the sheet-wide total isn't necessarily
complete just because one vendor's rows are.

Once every pending row for a vendor has been populated, that vendor's
total naturally becomes "Y/Y" and reads as complete — no special-casing
needed beyond the formula already excluding not-yet-populated rows.

### Native Excel constraints (no embedded code)

- **Data Validation** dropdown lists on every stakeholder/resolved score
  column, restricting entries to the project's existing scoring
  convention: 1.0, 1.5, 2.0, 2.5, 3.0, 3.5, 4.0, 4.5, 5.0.
- **Conditional formatting** (not a hard block) visually flags an
  unfilled score cell in a top-priority row — a nudge, not an
  enforcement mechanism, consistent with rows being optional.
- **Cell protection** (native Excel sheet/cell locking) on pending rows'
  stakeholder columns specifically — the one place this design uses a
  hard block rather than a nudge, since rating a row with no real
  evidence isn't a matter of taste.
- **Formulas only** for anything computed (rollups, subtotals, delta) —
  no macros, no add-ins.

### Audit trail substitute

Since this workbook lives outside git-tracked YAML once distributed,
`Resolved By` / `Resolved Timestamp` plus each stakeholder's own
column-set (which itself carries their identity in the header) serve as
the audit trail — "why did we pick this" is answerable by reading the
sheet directly. Combined with the `snapshot` command (below), this gives
a git-diffable point-in-time record even though live edits after that
point aren't continuously tracked.

## Collection modes

Both modes start from the exact same generated file — they differ only
in how multiple people's edits land in one place.

**Primary: shared live co-authored document.** The driver uploads the
generated workbook to whatever collaboration platform the org already
has (Microsoft 365 OneDrive/SharePoint co-authoring, Google Sheets, or
equivalent) and grants each stakeholder edit access to their own
column-set. They fill it in live (e.g. during a walkthrough meeting) or
asynchronously, at their own pace. There is no return trip and no merge
step — the file was never copied, so there's nothing to reconcile.

**Fallback: distributed copies + merge.** When live co-authoring isn't
available, the driver duplicates the generated file once per
stakeholder and distributes it however files already move around the
org (email, shared drive). Each stakeholder fills in their own
column-set in their own copy and returns it. The driver then runs
`merge` against each returned copy to fold it into the one master file.

## CLI command surface

New `dast-bench stakeholder-review` command group:

**`generate --vendor-id <id> [--vendor-id ...] --stakeholder <name:role
or :role> [--stakeholder ...] [--pending-criteria <vendor-id>:<criterion-id>[,<criterion-id>...] ...] --out <path>`**

Produces the one master workbook described above. Fails with a clear
`error:` message (relayed verbatim, per this project's existing
convention) if any named vendor has any criterion with no score at all —
this is a hard precondition, not a soft warning, since generating from
an incomplete paper pass would bake real gaps (not deliberate,
clearly-marked pending rows) into the workbook. `--pending-criteria`
marks specific criteria as pending for a specific vendor even though a
paper score already exists for them (see "Two rounds" above); it does
not affect the completeness check, since a pending criterion still has
a real paper score on file, it's just being deliberately withheld from
display.

**`populate --vendor-id <id> --file <path>`**

Round 2 only. Reads the given vendor's current scores for whichever
criteria were marked pending when that file was generated (tracked via
the hidden `criterion_id` column plus the placeholder marker already in
those cells), fills in the real score/evidence, removes the cell lock,
and clears the placeholder formatting — for that vendor's sheet only.
Every other cell in the file, including anything any stakeholder has
already entered anywhere, is left untouched. Safe to run once per vendor
whenever that vendor's `dast-scan` results land, independent of whether
other vendors in the same workbook are still pending.

**`merge --into <master-file-path> --from <returned-copy-path>`**

Fallback-mode only. For every stakeholder column-set known to the
master file, checks whether the returned copy has any non-blank entry
in that column-set:

- If present and valid (score in the allowed list; a `Dispute? = Y` row
  has a non-blank rationale) and the master's corresponding cell is
  blank: fill it in from the returned copy.
- If present and valid but the master's corresponding cell is *already*
  non-blank and differs from the incoming value: flag as a conflict,
  do not overwrite. The driver resolves it by hand, directly in the
  master file.
- If present but invalid (bad score value, missing required rationale):
  flag it, do not merge that cell.
- If the returned copy has a column-set that doesn't match any known
  stakeholder header on the master: flag it as unrecognized rather than
  guessing or silently absorbing it.
- Never touches a pending/locked row, regardless of what the returned
  copy contains for it (it should be locked in every legitimate copy,
  but `merge` treats it as out of bounds either way).

Prints a summary: cells merged, cells flagged invalid, conflicts, and
unrecognized columns. Safe to run once per returned copy, repeatedly, as
copies trickle back in over whatever timeframe.

**`validate --file <path>`**

Read-only lint pass, usable against either mode's file at any time — no
persistence, no mutation. Reports the same class of issues `merge`
flags (invalid score values, disputes without rationale, a resolved
score present without a matching `Resolved By`/`Resolved Timestamp`),
plus any pending row that's had its lock or placeholder tampered with —
without requiring a merge operation to trigger the check. Useful for
checking a live shared document's current state at any point.

**`snapshot --file <path> --vendor-id <id> [--label <text>]`**

Copies the current state of the given file into
`data/stakeholder-reviews-archive/<date>-<vendor-id>[-<label>].xlsx` —
git-tracked, same as every other data artifact in this project. This
command only stages the file; committing it follows the project's
existing pattern of the evaluator explicitly asking for a commit/push,
not an automatic side effect. Intended to be run right after `generate`
(a clean baseline), again after any `populate` (capturing the vendor's
now-complete state), and whenever else the driver wants a checkpoint —
each snapshot is an independent point-in-time copy; there's no
versioning or diffing between them built into the tool.

## Skill: `dast-stakeholder-review`

A new skill orchestrating the above, following this project's existing
pattern (skill wraps CLI with judgment, never mutates data directly):

1. Confirm scope — which vendors (any candidate with a complete paper
   score; check via `dast-bench status`/`dast-bench candidate list`),
   whether this round happens before or after a finalist/rejected
   verdict, and which criteria (if any, per vendor) are pending on
   upcoming `dast-scan` results.
2. Confirm the stakeholder roster (names/roles) for this round.
3. Compute and present the default priority order; let the driver adjust
   before generating.
4. Run `generate`.
5. Run `snapshot` immediately as a baseline.
6. Ask which collection mode applies, and guide accordingly — for the
   shared-doc mode, remind the driver to upload it and grant access; for
   the fallback mode, prompt for `merge` invocations as copies return.
7. Offer `validate` at any point the driver wants a status check.
8. As `dast-scan` results land for any pending vendor, run `populate`
   for that vendor, then `snapshot` again to capture the update.
9. Offer a final `snapshot` once the round is considered complete.
10. Summarize what was generated/merged/populated/snapshotted, so the
    driver has a clear record of what this invocation did.

## Testing approach

- Priority-ordering computation: given a taxonomy, a vendor's scores,
  and research-cache gap-check flags, assert the expected sort order
  (weight descending, low-score/gap-checked criteria pulled earlier
  within a weight band).
- `generate`: read the produced file back with `openpyxl` and assert
  expected sheet names, column headers, row count/order, the hidden
  `criterion_id` column, presence of data-validation objects on score
  columns, correct placeholder text/cell locking on rows marked pending,
  and presence of the expected formula strings in the rollup/delta
  cells, including the "X/Y available points" partial-completeness case.
- `populate`: fixtures covering — a pending row gets correctly filled
  and unlocked from current vendor data; non-pending rows and any
  stakeholder-entered cells are provably untouched before/after; running
  it against a vendor with no pending rows is a no-op, not an error;
  addressing works correctly even when the fixture sheet's rows have
  been reordered (proving `criterion_id`-based lookup, not row-position
  lookup).
- `merge`: construct small in-memory "master" and "returned copy"
  fixtures covering each case — blank-filled, valid conflict, invalid
  value, unrecognized column, multiple stakeholders present in one
  returned file, and a pending/locked row being ignored regardless of
  its content — and assert the reported outcome and resulting cell
  values match expectations for each.
- `validate`: fixtures with known injected issues (including a tampered
  pending-row lock), assert the reported list matches.
- `snapshot`: assert the file lands at the expected archive path with
  matching content.
