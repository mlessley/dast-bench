# Report Disclaimer & Source Hyperlinks — Design

## Purpose & Context

After reviewing the real regenerated `reports/`/`sample-report/` output, the
evaluator raised three related presentation-polish concerns:

1. **Draft/WIP framing.** Nothing in the generated artifacts currently
   signals "this is a demonstration of what the tool does," so a reader
   could mistake a scorecard or executive summary for a finished
   procurement recommendation ready to act on. The evaluator's words: "not
   like: 'hey, here is my final proposal'."
2. **Evidence source citations aren't clickable.** Evidence text
   frequently ends with a citation like `Source: stackhawk.com/blog/
   graphql-security`, rendered today as plain, non-linked text.
3. **General navigability.** Some cross-links between report artifacts
   (e.g. a vendor's name linking to its full scorecard) would help a
   reader drill in without scope-creeping into unnecessary linking for a
   small set of flat files.

All three are presentation-only changes over existing data — no new
fields, no new CLI commands, no change to what gets scored or how.

## Goals

- Every generated report artifact (`comparison-matrix.md`,
  `scorecard-<id>.md`, `dashboard.html`, `comparison-matrix.xlsx`,
  `executive-summary.md`) carries a clear draft/sample disclaimer near its
  top, generated by the tool itself so it survives every future
  regeneration automatically — never a manually-patched addition.
- A new `_linkify_sources(text: str) -> str` function in
  `core/render/markdown.py` detects bare domain/URL-shaped tokens
  anywhere in evidence text (not only immediately after the literal word
  "Source:") and converts each into a markdown link, applied to the
  Evidence column in `render_scorecard`.
- `dast-report`'s narrative links each vendor's name (in the "Where They
  Win" table and its own per-vendor section heading) to that vendor's
  `scorecard-<id>.md`.
- `dast-shortlist`'s research instructions gain a soft nudge to include an
  actual source URL when one exists, so future evidence keeps benefiting
  from the same linkification.

## Non-Goals / Out of Scope

- No new data fields, no changes to `Criterion`/`Vendor`/`ScoreEntry`.
- `comparison-matrix.md` does not gain an evidence column — it's
  deliberately compact (scores only); linkification only applies where
  evidence text is already shown (`render_scorecard`).
- No in-page anchor links from comparison-matrix criterion rows to their
  Scoring Legend entries (declined during brainstorming — the legend
  already sits directly above the table in the same file).
- No retroactive rewriting of existing evidence strings in
  `data/candidates/*.yaml` — linkification happens at render time, so all
  ~30 existing "Source:" citations benefit without touching the YAML.
- No feature-parity backfill for `dashboard.html`/`comparison-matrix.xlsx`
  beyond the disclaimer itself (they still lack the Weight
  column/Scoring Legend/Category Breakdown that markdown output already
  has — a known, separately-tracked gap from the earlier rendering-engine
  work, not addressed here).

## Architecture

Two tracks, matching this project's established split between
deterministic code and skill-authored judgment:

**Track A — code (`core/render/`):**

- `core/render/markdown.py`: a private module constant `_DOMAIN_TOKEN`
  (compiled regex) and a new function:
  ```python
  _DOMAIN_TOKEN = re.compile(r'\b((?:[a-zA-Z0-9-]+\.)+[a-zA-Z]{2,}(?:/[^\s,;]*)?)')

  def _linkify_sources(text: str) -> str:
      def replace(m):
          token = m.group(1)
          stripped = token.rstrip('.,;:)')
          trailing = token[len(stripped):]
          return f"[{stripped}](https://{stripped}){trailing}"
      return _DOMAIN_TOKEN.sub(replace, text)
  ```
  Verified directly against every real "Source:" citation currently in
  `data/candidates/*.yaml` (30 evidence/observation strings spanning
  clean single URLs, comma-separated multi-source citations, domains
  embedded mid-sentence in parentheses, and bare domains followed by
  trailing prose) plus four negative-case strings (a version number, a
  dotted version like `v4.0.1`, `e.g.`/`i.e.` abbreviations, and a
  regulation code like `23 NYCRR 500`) — every case produces the correct
  output with no false positives or malformed links.
- `render_scorecard` calls `_linkify_sources(entry.evidence)` before
  passing the evidence text through the existing `_md_cell` escaping (link
  syntax uses `[`/`]`/`(`/`)`, none of which `_md_cell` touches, so order
  doesn't matter for correctness — but linkify first, then escape, is the
  natural call order since escaping is the last step before insertion into
  the table).
- `render_scorecard` and `render_comparison_matrix` each gain a one-line
  disclaimer blockquote inserted immediately after their `#`-level title,
  before any other content.
- `core/render/html.py`'s `write_html` gains a styled `<div>` banner
  immediately after `<h1>DAST Tool Comparison Matrix</h1>`.
- `core/render/xlsx.py`'s `write_xlsx` gains a disclaimer as row 1 (single
  cell, plain text — no rich formatting available), a blank row 2, then
  the existing header row shifted to row 3.

**Track B — skill instructions (no code):**

- `.claude/skills/dast-report/SKILL.md`: Step 5's existing one-line
  skill-authored header note is extended (not duplicated) to also carry
  the draft/sample framing. Step 4 gains an instruction that vendor names
  in the "Where They Win" table and each per-vendor section's heading are
  rendered as markdown links to `scorecard-<id>.md` (a relative link,
  since both files live in the same directory whether that's `reports/`
  or `sample-report/`).
- `.claude/skills/dast-shortlist/SKILL.md`: Step 4 (Research each
  criterion) gains a soft nudge — when a specific, locatable source (a
  blog post, docs page, GitHub repo) exists, cite its actual URL so it
  benefits from render-time linkification; this is guidance, not a hard
  requirement, since some evidence is legitimately "inferred, low
  confidence" with no single citable source, and that's already a
  documented, acceptable pattern in existing data.

## Disclaimer Wording

The same core message, adapted per format, never abbreviated to the point
of losing the "not a final recommendation" framing:

> 🚧 Draft/sample output demonstrating what dast-bench produces — not a
> final vendor recommendation. Scores, weights, and evidence below are
> illustrative of a real evaluation in progress.

## Data Flow

1. `dast-bench render` (unchanged CLI entry point) calls `write_markdown`,
   `write_html`, `write_xlsx` exactly as today — each now additionally
   writes its disclaimer and (for markdown) applies `_linkify_sources` to
   evidence text, purely from data already loaded, no new I/O.
2. `dast-report`'s Step 5 write to `executive-summary.md` includes the
   strengthened header; Step 4's narrative composition includes the new
   vendor-name-to-scorecard links when writing the Where They Win table
   and per-vendor headings.
3. `dast-shortlist`'s Step 4 research process is unaffected in mechanism
   (still CLI-recorded via `record-score`) — only the guidance about
   citing locatable sources changes, affecting future evidence text
   content, not any code path.

## Error Handling

Nothing new. `_linkify_sources` has no failure mode — worst case on an
unanticipated future evidence string is a token that isn't actually a
working URL gets wrapped in a link (a false positive), which is a content
quality question for whoever authored that evidence, not a crash risk.
`rstrip('.,;:)')` only ever shortens a matched token, never raises.

## Testing

**Track A (code, pytest):**
- `_linkify_sources` unit tests covering: a single clean URL, a
  domain-only citation with no path, two comma-separated URLs (both
  linkified independently, comma preserved as plain text between them), a
  domain followed by trailing non-path prose (only the domain linkifies),
  a domain embedded mid-sentence inside parentheses with trailing
  sentence punctuation (`).`) correctly stripped from the link and
  preserved as trailing text, and the four negative cases (version
  number, dotted version, abbreviations, regulation code) confirmed to
  produce no link.
- `render_scorecard` gains an assertion that its evidence column renders
  a real evidence string's citation as a working markdown link.
- `render_scorecard`/`render_comparison_matrix`/`write_html`/`write_xlsx`
  each gain an assertion that the disclaimer text is present in their
  output.
- All existing tests in `tests/test_render_markdown.py` and
  `tests/test_render_html.py` continue to pass unmodified — both are
  substring/structural checks with no assertion on exact document
  position, so a prepended disclaimer doesn't break them.
- **`tests/test_render_xlsx.py` is the one exception**: its existing test
  asserts the header row is exactly `ws[1]` and the first vendor's first
  score is exactly `ws.cell(row=2, column=4)`. Inserting a disclaimer row
  (row 1) and a blank spacer row (row 2) shifts the header to row 3 and
  the first data row to row 4 — this existing test's row-position
  assertions must be deliberately updated to match (not left as a
  regression), since the row shift is an intended consequence of this
  design, not an accidental side effect. The final `Weighted Total` row
  assertion (`ws[ws.max_row]`) is already position-independent and needs
  no change.

**Track B (skill instructions, grep/diff-based, no pytest):** same
verification pattern as the earlier `dast-report` narrative-template
plan — confirm the strengthened header text and the new vendor-link
instruction are present in the rewritten `SKILL.md` files, and that
unrelated sections remain byte-for-byte unchanged.

**Real-world verification:** after both tracks are implemented, the true
acceptance test is re-running `dast-bench render` plus the `dast-report`
skill against real data and inspecting the actual output — same pattern as
the narrative-template work, not a task within either plan itself.
